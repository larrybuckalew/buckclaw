OVERVIEW
--------
Step 14 adds the post_message tool, which allows cron job agents to proactively
send messages back to a user's session without waiting for user input. This is
the critical missing piece that lets cron jobs communicate results.

KEY PROBLEM SOLVED
------------------
When a cron job runs on schedule, it fires in its own isolated session with no
user waiting at the terminal. Without post_message, the cron agent's response
has nowhere to go. This step provides the delivery mechanism.

TECHNICAL FLOW
--------------

1. User: "Message me in 5 minutes"
2. Agent reads session_id from Layer 4 runtime context
3. Agent creates CRON.md with target_session_id=<user_session>, one_off=true
4. CronWorker._tick() detects job is due after 5 minutes
5. CronWorker._dispatch_job() calls _ensure_cron_tools()
6. _ensure_cron_tools() registers mapping in context.cron_delivery_map
7. CronWorker publishes DispatchEvent to AgentWorker
8. AgentWorker._get_or_create_session() checks cron_delivery_map
9. If found, injects PostMessageTool pre-configured with user's session_id
10. Cron agent calls post_message("message text")
11. PostMessageTool.execute() publishes OutboundEvent(session_id=user_session)
12. DeliveryWorker picks up event and routes to user's channel
13. User receives message on their platform

KEY DESIGN DECISIONS
-------------------

Why post_message is only in cron sessions:
  - User-facing sessions have waiting user; don't need to post back
  - Cron sessions are headless; need a way to communicate
  - Keeps tool surface clean and intention-clear

Why one-off crons are the right primitive:
  - Persistence: survives app restarts
  - Scalability: thousands of jobs via single tick loop
  - Flexibility: handles both recurring and one-time tasks
  - Discoverability: all work visible as CRON.md files

Why deliver via OutboundEvent/DeliveryWorker:
  - Consistent with regular agent responses
  - Platform agnostic (Telegram, CLI, WebSocket, etc.)
  - Testable and mockable

TESTING EXAMPLE
---------------

1. Start app in CLI mode with a session
2. Ask: "Message me in 1 minute with 'test message'"
3. Wait 60+ seconds
4. Should receive "test message" in your session

Behind the scenes:
  - CronWorker detects the job
  - Injects PostMessageTool
  - Cron agent calls post_message()
  - Message delivered via existing pipeline
